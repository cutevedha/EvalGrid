Layers · two execution flows · the autonomous agent loop · file-by-file connections

1The 10-second mental model

One question — "is this AI output good?" — answered in two modes.

  • Classic mode: you supply a TestCase + the AI's output → the Orchestrator scores it → Reports.
  • Autonomous mode (Eval Agent): you supply a goal + a target system → the EvalAgent writes its own test cases, drives the target, scores outputs, adaptively red-teams weak spots, and emits a verdict → Reports.

Both share one engine: Orchestrator → Metric Registry → Evaluators → Guards → Scoring primitives.

3Flow A — Classic evaluation

The path Orchestrator.run(test_case, output) takes.

sequenceDiagram
    participant U as Caller
    participant O as Orchestrator
    participant D as deterministic
    participant S as semantic
    participant SF as safety - policy
    participant J as llm_judge
    participant G as pii + injection
    U->>O: run(test_case, actual_output)
    O->>D: exact / substring / regex
    D-->>O: scores
    O->>S: similarity / bleu / rouge
    S-->>O: scores
    O->>SF: policy_safe?
    SF-->>O: policy_safe
    O->>J: correctness / groundedness
    J-->>O: scores
    O->>G: detect PII / injection
    G-->>O: pii_found, injection_detected
    O->>O: compare vs thresholds
    O-->>U: EvalResult(passed, scores, notes)
  1. Input — a TestCase + the AI's actual_output.
  2. Deterministic — fast string/regex checks (deterministic.py → scoring.py).
  3. Semantic — meaning-based similarity (semantic.py, pluggable embedder).
  4. Safety — safety.py → guards/policy.py blocked-phrase check.
  5. LLM judge — real LLM if set_llm_client() was called, else heuristic.
  6. Guards — PII + prompt-injection detection.
  7. Gate — scores vs test_case.thresholds; PII is a hard fail.
  8. Output — an EvalResult.
The other evaluators (agent, rag, multiagent, performance, robustness, bias, embedded, toxicity, hallucination) register into the Metric Registry on import and run on demand via orchestrator.compute_metric("name", …) — not in the default run() path.

4Flow B — Autonomous Eval Agent

The headline feature: EvalAgent.run(goal, capabilities).

        RECORD["RECORD in AgentMemory"] --> REFLECT
        REFLECT["REFLECT: keep probes

    end
    REFLECT --> ROUNDLOOP
    ROUNDLOOP -->|no weak probes
or budget spent| BUILD
    BUILD["Build EvalReport"] --> SUMM["Summarise verdict
(LLM polish optional)"]
    SUMM --> OUT([EvalReport: verdict, findings, results])
#PhaseFileWhat happens
1Planagent/planner.pyGoal + capabilities → ProbeSpecs. Capability words add functional probes; risk words add red-team probes from synthetic/redteam.py.
2Generateagent/agent.pyRound 1 = base inputs. Later rounds mutate failing inputs into harder variants (redteam + augmentation).
3Executeagent/target.pyEvalTarget calls the SUT (LLM / callable / offline). Errors become "Error: …".
4Evaluatecore/orchestrator.pyScored by the Orchestrator; adversarial probes add a refusal score (classifier → LLM judge → keyword).
5Recordagent/memory.pyStores results per probe; computes pass rates + weakest metric.
6Reflectagent/agent.pyProbes ≥ 0.8 pass are dropped (settled); weak ones survive. Loop ends when none remain.
7Reportagent/report.pyRolls memory into ProbeFindings + an EvalReport verdict + summary.
Adaptive narrowing (steps 2 & 6) is what makes it autonomous: budget concentrates on real weaknesses instead of re-testing what already passes.
